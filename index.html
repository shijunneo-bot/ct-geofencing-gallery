The limits. Up to 100 geofences in each cluster and up to 100 clusters per app. A national store network fits in one cluster; a regional one fits in a handful.
The events. Cluster-level Geofence Cluster Entered and Exited, plus fence-level entry events, usable across campaigns, segmentation, and analytics. The geocluster name appears under event properties once it has fired at least once.
The payload. Every event carries id, gcId, gcName, lat, lng, r (radius in metres), and the triggered coordinates. The fence id tells you which store, branch, or terminal, which is what makes one cluster serve a whole network.
Quiet hours are native. Each cluster carries a tracking schedule (start and end times, including future dates) and Do Not Disturb days and times. Venue fences track on event days; branch fences track in branch hours.
Accuracy honesty. Phone positioning is typically accurate to 20-50 metres. Fences sized under that are wishful; the cards here size fences for the technology, not the floor plan.
Permission is the product. The SDK will not request location permission; the app must, at a moment where the value is obvious. Every play in this guide depends on that opt-in, so the ask deserves design attention.
Plan gating. Geofencing ships with the Advanced Plan. Worth confirming entitlement per account before the workshop, not after it.
Dwell is free. Entry-to-exit time is a built-in dwell measure: it filters drive-bys, times workouts, and qualifies visits without any extra instrumentation.
Where the other guides plug in: the fence supplies the moment; Linked Content (FG06) supplies live truth inside the message (queue depth, room-ready status, surge rates); Recommendations (FG07) supplies the picks for what-next plays. The strongest cards below run two or three guides at once.

Integration checklist (hand this to engineering)

  1. Integrate the CleverTap Geofence SDK alongside the main SDK: Android uses the Play Services Location library, iOS uses Core Location.
  2. Design the permission ask in the app: the SDK never requests location permission, and the opt-in moment decides the reachable audience for every play in this guide.
  3. Draw clusters to match operations: fence radius at or above realistic accuracy (20-50m), tracking schedules for venue and branch hours, DND for the hours nobody wants a push.
  4. Use the fence id in the event payload to personalize per site (which store, which branch, which terminal) from a single cluster.
  5. Add conditions on top of fences: active PNR, order-ready status, renewal window, on-duty state. The fence catches everyone; the condition chooses who hears.
  6. Frequency-cap generously. Fences at malls, airports, and transit hubs fire constantly; the caps in these cards are load-bearing, not decorative.

Honest edges

Location permission opt-in rates are the ceiling on every play here, and battery-usage perception is real even when the SDK is efficient, so lead with visible value. Accuracy of 20-50 metres means doorway-level precision is not promisable. And Geofencing is an Advanced Plan capability, so entitlement is a pre-sales check, not a post-sales surprise.

The developer-facing half of this guide: how to stand up the CleverTap Geofence SDK on both platforms, design the permission ask the SDK deliberately does not make, draw clusters in the dashboard, qualify campaigns off the events, and read the payload. Written to be handed to an engineering team as-is, or walked through on a TAM call. 🤖 2 · Android SDK enablement Play Services Location▶
1
Prerequisites. The main CleverTap Android SDK must already be configured, and the project set up for Google Play Services. The Geofence SDK is an add-on, not a replacement.
2
Add the dependencies to build.gradle. The work-runtime and concurrent-futures entries are required for the periodic last-location fetch mode:
implementation "com.clevertap.android:clevertap-geofence-sdk:1.3.0"
implementation "com.clevertap.android:clevertap-android-sdk:6.2.1"  // 3.9.0+
implementation "com.google.android.gms:play-services-location:21.0.0"
implementation "androidx.work:work-runtime:2.7.1"              // FETCH_LAST_LOCATION_PERIODIC
implementation "androidx.concurrent:concurrent-futures:1.1.0"  // FETCH_LAST_LOCATION_PERIODIC
3
Manifest permissions ship with the SDK. The location permissions the SDK needs are already declared in its own manifest and merge in automatically; the app adds nothing to AndroidManifest.xml. Requesting them at runtime, however, is entirely the app's job (section 4).
4
Initialize after CleverTapAPI. CTGeofenceAPI takes the CleverTapAPI instance, so order matters. Configure behaviour through CTGeofenceSettings: location accuracy, fetch mode, logging level, geofence count, plus tuning like setSmallestDisplacement() (metres) and setGeofenceNotificationResponsiveness() (milliseconds). The API activates on init().
5
Trigger a location when it helps. CTGeofenceAPI.getInstance(context).triggerLocation() fetches the last known location from the OS (can be null) and delivers it through CTLocationUpdatesListener. Useful right after permission is granted, so fence setup does not wait for the user to move.
6
Listeners for control. The SDK provides main-thread callbacks: geofence API initialized, location updates, and geofence enter/exit, for any custom in-app behaviour on top of the automatic events.
7
The events are SDK-raised, full stop. Geofence Cluster Entered and Geofence Cluster Exited fire automatically; the app cannot raise them manually. This is what keeps the event stream trustworthy enough to trigger campaigns from.
TAM talking point: the two Android settings worth naming on a technical call are smallest displacement and responsiveness: they are the battery-versus-latency dials, and the fact they exist is usually enough to settle the battery objection. Minimum Android version is 5.0 (API 21) on current SDK releases; check the changelog if the account supports older fleets.
🍎 3 · iOS SDK enablement Core Location▶
1
Install via CocoaPods, Swift Package Manager, Carthage, or manually. CocoaPods:
pod 'CleverTap-Geofence-SDK'
pod 'CleverTap-iOS-SDK'
2
Enable Background Modes → Location Updates on the App Target in Xcode. Without it, fences only work while the app is foregrounded, which defeats most of this guide.
3
Add the Info.plist usage strings. NSLocationAlwaysAndWhenInUseUsageDescription (and the When-In-Use key) carry the text iOS shows in the permission dialog. This string is marketing copy wearing a plist key; write it like the ask it is.
4
The SDK will NOT request location permission. Core Location region monitoring starts only after the app has obtained permission, at a moment the app chooses. This is a design position, not a gap (section 4).
5
Start the Geofence engine after the main SDK initializes, and set CleverTapGeofenceLogLevel before invoking start if you want setup visibility in the console.
6
Optional: subscribe to the notifications. CleverTapGeofenceEntered and CleverTapGeofenceExited fire as NSNotifications on fence transitions, for custom in-app actions (unlock a screen, refresh a view) on top of the automatic dashboard events.
TAM talking point: on iOS the permission ladder is the conversation: When-In-Use covers foreground plays, Always is what background fence detection wants, and iOS grants Always via a provisional flow the OS controls. Set expectations that the reachable audience is "users who granted Always", and design the ask accordingly. The Background Modes checkbox in step 2 is the most commonly missed step in the field.
🔐 4 · The permission flow the app asks, the SDK consumes▶

Both platform SDKs share one rule: the SDK never asks for location permission; the app must. This puts the highest-leverage decision of the whole integration in UX, not code. The permission dialog is a one-shot conversion surface, and every card in the gallery tab lives downstream of it.

1
Ask at a value moment, not at launch. A food app asks when the user taps "find outlets near me"; a banking app asks when they open the branch locator. Launch-time asks convert worst and are the hardest to recover from.
2
Pre-permission screen first. Explain the value in the app's own UI before invoking the OS dialog, because the OS dialog is unskippable ammunition: decline it hard on Android or iOS and the recovery path runs through system settings.
3
Android specifics. Runtime location permission is required for the SDK to work; on Android 10+ background location ("Allow all the time") is a separate, later grant the OS itself gates. Foreground-first, background-later is the platform-sanctioned sequence.
4
iOS specifics. The usage-description strings from section 3 are the ask's copy. When-In-Use vs Always determines whether background fence plays work at all.
5
Right after the grant, call triggerLocation() (Android) or let the engine take its first fix, so the user's fences arm immediately instead of on the next commute.
TAM talking point: permission opt-in rate is the ceiling on every geofencing KPI, so it belongs on the dashboard next to the campaign metrics. When an account says "geofencing isn't performing", the first diagnostic question is "what share of the audience has location permission granted?", and it is remarkable how often nobody knows.
🗺️ 5 · Cluster creation in the dashboard Settings → Geofences▶
1
Navigate to Settings → Geofences, click Add Cluster, and select Create.
2
Pick the unit of measure (kilometres or miles) for tracking, then continue.
3
Search the location on the map or via the search bar, then in the settings pop-up give the geofence a name and set its radius. Save, and repeat for every fence the cluster needs: up to 100 geofences per cluster, and up to 100 clusters per app.
4
Name the cluster and set its tracking window: start and end times for when the cluster tracks users at all, including future dates. A sale-weekend fence can exist Friday-to-Sunday and never fire on the setup crew.
5
Set DND days and hours per cluster. Quiet hours are native to the cluster, not something every campaign must remember: branch fences track in branch hours, venue fences on event days, and nobody gets a 2am fence push because one campaign forgot.
6
Radius honesty. Practical positioning accuracy runs 20-50 metres. A 150m fence catches a doorway reliably; a 30m fence is wishful. Size fences for the technology, not the floor plan.
TAM talking point: the cluster-level schedule and DND settings are the most under-used part of this product, and they are the answer to the creepiness objection: "the fence literally does not track outside the hours you choose" lands better than any privacy-policy citation. The 100×100 limits translate to "a national store network is one cluster", which reframes the cap as headroom.
🎯 6 · Campaign qualification off the events Entered / Exited as triggers▶
1
The two events are Geofence Cluster Entered and Geofence Cluster Exited, raised automatically by the SDK. They behave like any other CleverTap event: campaign triggers, Journey entry points, segmentation, funnels, analytics.
2
Real-time trigger: create a campaign (push is the classic) with "when a user does" → the geofence event, optionally filtered on event properties like gcName to scope it to one cluster.
3
Property filtering: the geocluster name appears under the event's properties once the event has fired at least once, so build the fence, walk it (or wait for traffic), then wire the campaign filter.
4
Layer segment conditions on top. The fence catches everyone who crosses it; the campaign chooses who hears. Open pickup order, renewal window, on-duty state, tier: the strongest cards in the gallery are all fence + condition, never fence alone.
5
Journeys unlock the rest: dwell (Entered → wait → check no Exited), follow-ups (Entered → wait until next morning), and abandonment patterns (Exited → check missing event). Every non-real-time card in the gallery is one of these three shapes.
6
Delivery is channel-independent. The event fires from the device, but the responding campaign can go out on push, in-app, SMS, WhatsApp, email, or a webhook. Half the ops plays in the gallery deliver to a system, not a person.
TAM talking point: "geofence campaign" undersells it; the accurate sentence is "the fence is just an event, and everything CleverTap does with events now works on location." That reframe is what moves an account from one walk-in-welcome campaign to a portfolio of dwell, exit, and pattern plays.
📦 7 · Payload reference the SDK event schema▶ 
{ "id": 500043,            // fence id: WHICH store / branch / terminal
  "gcId": 1,               // cluster id
  "gcName": "SG Stores",   // cluster name, Liquid-addressable in copy
  "lat": 1.3006,           // fence centre
  "lng": 103.8368,
  "r": 150,                // fence radius, metres
  "triggered_lat": 1.3004, // where the user actually was at trigger
  "triggered_lng": 103.8371 }

The two workhorses: id is what lets one cluster serve a whole network while every message knows which outlet fired it, and gcName is the free personalization token; half the Liquid cards in the gallery are just {{ gcName }} doing honest work. The triggered_* pair vs the fence centre is your accuracy audit: consistently large gaps on one fence mean the fence is drawn wrong or the radius is fighting the environment.

TAM talking point: when an account asks "can we tell which store?", the answer is in this payload, not in more clusters. One cluster, 100 fences, and id does the disambiguation, which is materially simpler than the fence-per-campaign architecture accounts sometimes invent for themselves.
🧠 8 · The hacks page field-earned patterns▶

Dwell vs enter

There is no native dwell event; dwell is a Journey shape: Entered starts a wait, Exited cancels it, and surviving the wait IS the dwell. The threshold is the design decision: 20 minutes separates shopping from passing through, 45 separates exploring from a bus ride, 2 hours separates a hospital day from a visit. Entry plays create reach; dwell plays create qualification. When a card feels spammy as an entry trigger, it is usually a dwell play wearing the wrong mechanic.

TAM talking point: dwell needs no extra SDK work, no extra product, and no extra cost, just a Journey. "You already own dwell" is a genuinely fun thing to tell an account.

Cluster hygiene at the 100-fence cap

The cap is generous until it is not. The habits that keep it generous: one operational network per cluster (stores in one, pop-ups in another, never mixed, because DND and schedules are cluster-level); a naming convention from day one (SG-Stores, SG-Popup-2026Q3) because gcName goes into Liquid copy and analytics forever; archive temporary clusters when the event ends rather than letting sale-weekend fences haunt the account; and audit fences quarterly against the actual outlet list, since closed stores with live fences are the classic silent rot.

TAM talking point: a cluster audit is a 30-minute QBR agenda item that reliably surfaces value: dead fences, unfenced new outlets, and schedule gaps. It positions the CSM inside the account's operations, which is where we want to live.

Battery and accuracy tradeoffs

The SDK rides the platform geofencing services, not continuous GPS, so the baseline is efficient. The dials, when someone wants them: Android's accuracy setting, smallest displacement, and notification responsiveness trade freshness for battery explicitly. Practical accuracy is 20-50m regardless of settings, which drives two rules: radius ≥ 100m for anything that must fire reliably, and never promise doorway-level precision. Urban canyons, basements, and underground stations degrade further; prefer exit triggers at street level over entry triggers underground (the transit cards in the gallery all do this deliberately).

TAM talking point: when a client asks for a 30m fence on a shopfront, the counter is a 120m fence plus a campaign condition, which fires reliably and filters precisely. Accuracy problems are usually solved in qualification, not radius.

Testing fences before launch

A repeatable test protocol: (1) a staging cluster around the office, radius 150m, so every build can be tested on foot; (2) mock-location tools on Android emulators and Xcode's simulated locations for desk testing transitions; (3) triggerLocation() after permission grant to confirm the pipeline without moving; (4) verify the event lands in the dashboard with the right gcName and properties before wiring any campaign; (5) one real walk test per fence type you deploy, because emulators do not model urban GPS. And test the DND window by trying to trigger inside it: silence is a pass, not a bug.

TAM talking point: the walk test is a genuinely good joint activity for a workshop day: fence the client's own office, walk out for coffee together, watch the event arrive. Nothing sells the mechanic like seeing your own dot cross your own line.
